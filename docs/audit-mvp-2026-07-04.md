## 1. Was tracken wir aktuell?

Aus dem Asset-Katalog (`internal/marketdata/mock.go` = `RegistryProvider`):

| Kategorie | Anzahl | Quelle |
|---|---:|---|
| Einzelaktien | **100** | Finnhub (`/quote`) |
| ETFs | **15** | Finnhub |
| Rohstoffe (via ETF-Proxy GLD/SLV/USO/…) | **6** | Finnhub |
| Krypto | **8** | CoinGecko (`/simple/price`) |
| **Summe handelbarer Katalog** | **129** | |
| eSports (LoL) | dynamisch | lolesports-Schedule + Polymarket-Odds |

eSports ist nicht Teil der 129 — es sind pro Match/Team generierte Event-Märkte (Yes/No-Kontrakte), gefiltert nach Liga (LEC, LCK, LCS …). Team-Fallbackliste ist im Code hardcodiert (`esports.go`).

> ⚠️ **Ohne `FINNHUB_API_KEY` haben alle 121 Nicht-Krypto-Assets Preis 0** (der Registry-Provider liefert nur Namen/Symbole; echte Preise kommen ausschließlich vom Finnhub-Overlay). Das ist der Default-Zustand (`MARKET_DATA_PROVIDER=mock`). CoinGecko braucht keinen Key, daher zeigen die 8 Kryptos Preise.

## 2. Ist die Markt-API robust genug zum „einfach mehr Aktien hinzufügen"?

**Kurz: Die Absicht ist richtig, die Umsetzung ist aktuell kaputt.** Der Round-Robin-Poller (`pollSingle`, 1 Asset alle 3 s) ist ein sauberes Staggering-Konzept — aber vier Dinge zerstören es:

### 2a. 🔴 P0 — `Markets()` löst einen Full-Fetch aller Finnhub-Quotes aus
`Service.Markets()` → `FinnhubProvider.Markets()` iteriert **sequenziell über alle 121 Symbole** und macht pro Symbol einen `/quote`-Call (`finnhub.go:177-201`). `Markets()` wird aufgerufen von **4 Stellen**:
- `handlers.go:59` — jedem `GET /api/markets` (jeder App-Load)
- `history.go:102` — jedem Chart-Load (Preis-Anker)
- `sync.go:140` — jedem Portfolio-Sync
- `poller.go:72` — **alle 3 Sekunden** (nur um die Asset-ID-Liste zu lesen!)

Folgen mit gesetztem Finnhub-Key:
- Finnhub Free-Limit = 60 req/min. Ein einziger App-Load = 121 Calls → **sofortiges Rate-Limit**.
- Der 3-s-Poller feuert alle 3 s 121 Calls → dauerhaft weit über Limit.
- 121 sequentielle Calls à 5 s Timeout können die 30-s-Request-Deadline (chi `middleware.Timeout`) reißen → `/markets` fällt auf Registry zurück → **zeigt Preis 0**. D. h. gerade *mit* Key wird es oft schlechter.

**CoinGecko** hat dasselbe Problem in milder Form: `CoinGeckoProvider.Markets()` fetcht **immer** (kein Key-/TTL-Guard, `coingecko.go:105-138`) alle 8 Coins bei *jedem* `Markets()`-Call → jeder App-/Chart-/Sync-Load geht live gegen CoinGecko (Demo-Limit ~30/min).

**Empfehlung (Architektur):** Katalog und Preis-Anreicherung strikt trennen.
- `Markets()`/Handler dürfen **nur** aus DB/Registry lesen (billig, kein Provider-Call).
- Nur der Poller ruft Provider-Quotes ab und schreibt sie in die DB (`asset_quotes`).
- Preise in Handlern kommen aus `LatestQuotes` (macht `marketdata.go` teilweise schon — aber `Markets()` überschreibt sie mit dem Live-Overlay).

### 2b. 🟠 P1 — `MarketDataRefreshWindowSecs` (900 s) ist toter Code
`config.go:16` dokumentiert ein Refresh-Fenster, über das gestaffelt wird — der Poller nutzt aber ein **hartkodiertes 3-s-Ticker** (`poller.go:14`) und ignoriert die Config. Bei 129 Assets = 387 s/Zyklus. Fügt man Assets hinzu, wird der Zyklus linear länger, ohne dass man das steuern kann. Der Poller sollte das Intervall aus `RefreshWindow / len(assets)` berechnen.

### 2c. 🟠 P1 — Assets hinzufügen erfordert Edits an 2 Stellen, keine dynamische Registrierung
Ein neues Asset muss in **`mock.go`** (Fixture) *und* **`finnhub.go`** (`assets`-Map Symbol-Mapping) eingetragen werden. Kein Admin-UI, kein Config-File, keine DB-getriebene Asset-Verwaltung. Für „immer mehr Aktien" ist das der eigentliche Reibungspunkt. Empfehlung: einen einzigen Asset-Katalog (JSON/DB) als Single Source of Truth, aus dem sich Fixture *und* Symbol-Mapping ableiten.

### 2d. 🟡 P2 — Batching ungenutzt
Finnhub `/quote` ist single-symbol (ok), aber CoinGecko `/simple/price` unterstützt Multi-ID-Batching — der Poller fragt trotzdem 1 Asset/Tick ab. Für Krypto könnte 1 Call alle 8 Coins holen.

**Bewertung der Ausgangsfrage:** Der Poller *staggert* korrekt (nie mehr als ~20 req/min *aus dem Poller selbst*), aber die `Markets()`-Overlays laufen daran vorbei und sprengen das Limit. Nach Fix 2a skaliert das Design sauber auf hunderte Assets (nur der Refresh-Zyklus wird länger — Fix 2b macht das steuerbar).

## 4. Kritische Bugs (P0 — Blocker vor Veröffentlichung)

### 4.1 🔴 History wird bei jedem Server-Neustart gelöscht
`sqlite.go:135`: `DROP TABLE IF EXISTS asset_history;` läuft **unbedingt in `configure()`** bei jedem Start. Das ist als einmalige Migration für das neue tiered-Schema gedacht, wird aber jeden Neustart ausgeführt → **alle akkumulierten Kursverläufe weg**. Muss durch eine versionierte/idempotente Migration ersetzt werden (z. B. Spalte prüfen, `ALTER`, oder Migrationsversion in einer Meta-Tabelle).

### 4.2 🔴 eSports-Wetten sprengen den Portfolio-Sync
`sync.go:175-189` validiert jede Position/Transaktion gegen `knownAssets` (den Markt-Katalog). eSports-Positionen haben `assetId = event:lol:<matchId>:<team>` und sind **nicht** im Katalog (0 Event-Assets in den Fixtures). Ergebnis: Sobald ein Nutzer **eine** eSports-Wette hält, schlägt `PUT /api/sync/portfolio` mit `unknown asset id` (400) fehl → **das gesamte Portfolio synchronisiert nie mehr**. Zusätzlich: `portfolio_positions.asset_id` hat einen FK auf `assets` (`ON DELETE RESTRICT`), der Insert würde also ohnehin scheitern.
Trigger-Pfade: Login-Auto-Sync (`handleUserLogin`), Register (`handleUserRegister`), Cloud-Sync-Button.
**Fix:** Event-Assets beim Wetten in `assets` upserten *oder* Event-Positionen in der Sync-Validierung als Sonderfall zulassen (Regex `^event:` statt Katalog-Lookup).

### 4.3 🔴 Charts sind beim Launch leer (kein History-Backfill)
History kommt jetzt ausschließlich aus akkumulierten Poll-Daten in SQLite. Frische Instanz = **keine** Datenpunkte. 1D braucht 24 h Laufzeit, 1W eine Woche, **1Y ein Jahr**, bis der Chart gefüllt ist. Der Poller schreibt zudem nur 1 Asset/3 s → pro Asset alle ~6,5 min ein Punkt. Für einen „veröffentlichbaren" Eindruck fehlt ein Backfill (Finnhub `/stock/candle` bzw. CoinGecko `/market_chart`) oder zumindest eine ehrliche „noch keine History"-Leerdarstellung statt eines leeren Charts.

### 4.4 🔴 `Markets()`-Rate-Limit-Kaskade
Siehe 2a. Ohne Fix ist die App mit echtem Finnhub-Key faktisch unbenutzbar (Timeouts, Preis 0).

### 4.5 🟠 P0/P1 — Klick auf eSports-Position im Portfolio zeigt falsches Asset
`selectMarket(position.assetId, true)` bei einer Event-Position setzt `selectedAssetId`, aber `selectedMarket = markets.find(...) ?? markets[0]` fällt auf **markets[0]** zurück (Event ist nicht im Katalog). Der Nutzer landet im Trade-View auf z. B. AAPL, ohne Hinweis. Entweder Event-Positionen im Portfolio nicht auf Trade verlinken, oder einen eSports-Detailpfad bauen.

## 5. Null-Guards & Race Conditions

- 🔴 **`selectedMarket` ungeschützt** in reaktiven Statements: `App.svelte:262-266` (`chartHigh/Low/Open/rangeChange`) und `:889/:892` (Landing-Preview) greifen direkt auf `selectedMarket.priceCents/.symbol` zu. Wenn `fetchMarkets()` fehlschlägt (Backend down), ist `markets=[]` → `selectedMarket=undefined` → **Runtime-Fehler / weißer Screen** statt Fehlerbanner. Guard einbauen (`selectedMarket?.priceCents ?? 0`).
- 🟠 **`historyToken` deckt Chart ab, aber nicht Quotes:** `refreshQuotes` (alle 30 s) und `loadHistory` können konkurrieren; Quotes-Race ist unkritisch (idempotent), aber `markPositionsToMarket` + `savePortfolio` laufen aus mehreren Triggern (Quote-Poll, eSports-Reconcile, Settle) ohne Serialisierung → letzter `savePortfolio`-Write gewinnt, potenziell mit veraltetem Snapshot (Lost Update auf IndexedDB).
- 🟠 **Sync-Konfliktauflösung ist Last-Write-Wins per `updatedAt`** (`restoreSyncedPortfolio`). Zwei Geräte → stilles Überschreiben ohne Merge/Warnung.
- 🟡 **Gleicher `clientId` = geteiltes Portfolio:** Portfolio-ID = `sha256(clientId:portfolioId)`. Zwei Nutzer im selben Browserprofil (vor Login) teilen sich denselben Datensatz.
- 🟡 **`settleResolvedBets` / `reconcileEsportsPositions`** modifizieren `portfolio` async und speichern; wenn parallel ein Trade läuft, kann eine der beiden Versionen verloren gehen.

## 6. UI/UX-Audit

**Stark:** Klares Trading-Cockpit, Tastenkürzel (B/S/1-6/?), Skeleton-Loader, Toasts, saubere Money-/BPS-Formatierung, responsives Grid, Dark-Terminal-Look, Equity-Kurve mit realem/ unrealem P&L und Drawdown.

**Schwächen / Findings:**
1. 🟠 **Preis-0-Darstellung inkonsistent:** Watchlist nutzt `formatPrice` (zeigt „—"), aber Markets-Grid-Cards (`App.svelte:1252`) nutzen `formatMoney` → zeigen **„$0.00" + „0.00%" + Fake-Sparkline** für ungepreiste Assets. Wirkt kaputt.
2. 🟠 **Order-Typen Limit/Stop sind Fassade:** Es gibt Limit-/Stop-Auswahl, aber keine Order-Verwaltung — eine „Limit"-Order wird **sofort** zum eingegebenen Preis ausgeführt (kein Warten aufs Erreichen). Entweder echte pending Orders bauen oder Limit/Stop entfernen, sonst irreführend.
3. 🟡 **Event-Trade-Panel im Trade-View ist toter Pfad** (kein Event im Katalog, `isEvent` nie true über Markt-Auswahl). Der ganze Yes/No-Order-Zweig ist praktisch unerreichbar außer über eSports-View.
4. 🟡 **Landing-Stat „{markets.length} Märkte"** zählt 129 auch wenn 121 Preis 0 haben → optimistisch.
5. 🟡 **Status-Pill „API online / Local mode"** basiert nur auf `config`-Präsenz, nicht auf echter Datenverfügbarkeit → sagt „online", während Preise 0 sind.
6. 🟡 Keyboard-Shortcut „1-6" wählt `markets[0..5]` (sortiert alphabetisch nach Symbol) — nicht zwingend die 6 in der Market-Tape sichtbaren, aber deckungsgleich weil beide `markets.slice(0,6)` — ok.
7. 🟡 Kein „source"-Zeitstempel-Alter sichtbar (Nutzer sieht nicht, ob ein Preis 3 s oder 3 h alt ist).

## 8. Simulierte User-Workflows & Edge Cases

1. **Neuer Nutzer, kein Finnhub-Key (Default-Deploy):** öffnet App → 121 Assets zeigen „—"/„$0.00", nur 8 Kryptos handelbar, Charts leer. **Wirkt kaputt.** → braucht Key + Backfill oder ehrliche Kommunikation.
2. **Nutzer platziert eSports-Wette, loggt sich ein:** Auto-Sync 400 → Toast „Sync fehlgeschlagen", Portfolio bleibt lokal. Verwirrend. (Bug 4.2)
3. **Nutzer kauft AAPL, Preis aktualisiert sich, Reload:** Position ok (IndexedDB), Preis via Quote-Poll ok. ✅
4. **Backend offline beim Laden:** weißer Screen möglich (Bug 5, `selectedMarket` undefined) statt Fehlerbanner.
5. **Zwei Tabs offen:** beide pollen Quotes & schreiben Portfolio → mögliches Lost Update in IndexedDB.
6. **Server-Neustart (Deploy):** komplette Preis-History weg (Bug 4.1).
7. **Limit-Order „unter Marktpreis" kaufen:** wird sofort zum Limit ausgeführt (nicht realistisch, Bug 6.2).
8. **eSports-Match läuft, Wette gehalten, Poller settlet:** `settleResolvedBets` zahlt 100¢/0¢ — funktioniert; aber Auszahlung basiert auf `winnerCode`-Vergleich (Case-insensitiv, ok). Wenn `parseEsportsAsset` fehlschlägt (Format-Drift), Position bleibt für immer offen.

- 🟡 Default Finnhub-/CoinGecko-Keys leer → stiller Fallback auf Preis 0 ohne Betreiber-Warnung (kein Health-Signal „Preise stale").
